# Getting Started: Demo, Runtimes, Dashboard, and Customization

This guide is the practical walk-through for learning `malaclaw` end to end.

It is written around one starter, `podcast-production-pipeline`, so you can test the same project in multiple runtimes, inspect it in the dashboard, and then customize it in small steps.

## First: Should `starter init` copy team and agent templates into the project?

Short answer: not by default.

Current behavior is intentional:

- `starter init` creates a lightweight project scaffold
- the project owns `malaclaw.yaml`
- the repo keeps owning the bundled packs, teams, agents, and skills
- `install` resolves the manifest against those templates and provisions the runtime files

This keeps starter projects small and lets them pick up improvements from the bundled templates.

What the current model is missing is a better per-project customization entry point. Today, if you want to customize team structure, topology, or agent definitions, the supported path is to create an overlay templates directory and point `MALACLAW_TEMPLATES` at it.

That means:

- no, `starter init` does not need to copy all templates by default
- yes, a future convenience flow would be useful

Good future options would be:

- `malaclaw customize init` to scaffold only the team and agent files you want to override
- or `malaclaw starter init --with-overlay` to create a project-local overlay directory

Until that exists, the guide below shows the manual overlay workflow.

## Three Ways to Start a MalaClaw Project

You do not have to start from a demo starter.

There are three real entry paths:

### A. Start from a demo starter

This is the easiest learning path.

Use it when:

- you want a known-good project shape
- you want to see a complete example quickly
- you want the docs and demo cards to match what you are testing

Example:

```bash
mkdir -p /tmp/malaclaw-guided-demo
cd /tmp/malaclaw-guided-demo
malaclaw starter init podcast-production-pipeline ./podcast-openclaw
cd ./podcast-openclaw
malaclaw install --dry-run
```

This creates:

- `malaclaw.yaml`
- `STARTER.md`
- `DEMO_PROJECT.md`

### B. Start from an existing repo

Use it when:

- you already have a repo
- you want to manage a real workflow instead of a demo
- you want to choose packs and skills interactively

Step by step:

```bash
cd /path/to/existing/repo
malaclaw init
```

`malaclaw init` is an interactive wizard. It does not create an empty file. It asks you which packs and optional skills to include, then writes a real `malaclaw.yaml`.

The generated manifest is prefilled with:

- `version: 1`
- `runtime: openclaw`
- `project.id` derived from the directory name
- `project.name` derived from the directory name
- the packs you selected in the wizard
- the optional skills you selected in the wizard

Then continue with:

```bash
cat malaclaw.yaml
malaclaw install --dry-run
malaclaw install
malaclaw doctor
```

So yes, starting from an existing repo still begins with a new `malaclaw.yaml`, but not an empty one. It is generated by the wizard as the project's initial desired state.

### C. Start from OpenClaw first, without a project manifest

Use it when:

- you want OpenClaw to guide the setup first
- you are not ready to formalize a managed project yet
- you want `malaclaw-cook` installed into OpenClaw first

Step by step:

```bash
mkdir -p /tmp/malaclaw-bootstrap-only
cd /tmp/malaclaw-bootstrap-only
malaclaw install
```

If there is no `malaclaw.yaml`, MalaClaw does zero-config bootstrap instead of failing.

That path:

- installs the bundled `malaclaw-cook` skill into the main OpenClaw workspace
- updates the main OpenClaw guidance files
- leaves you in the OpenClaw-first workflow until you choose to create a managed project later

This path does not start with a project manifest. The project manifest comes later only if you choose to promote the work into a managed project.

## 1. Start with one demo project in OpenClaw

Create a clean test workspace:

```bash
mkdir -p /tmp/malaclaw-guided-demo
cd /tmp/malaclaw-guided-demo
malaclaw starter init podcast-production-pipeline ./podcast-openclaw
cd ./podcast-openclaw
```

You should now have:

- `malaclaw.yaml`
- `STARTER.md`
- `DEMO_PROJECT.md`

What they are:

- `malaclaw.yaml`: the real project manifest
- `STARTER.md`: generated quick summary and next steps
- `DEMO_PROJECT.md`: richer demo card copied from the catalog

Inspect the starter metadata:

```bash
cat malaclaw.yaml
malaclaw starter show podcast-production-pipeline
malaclaw team show content-factory
```

Preview the install:

```bash
malaclaw install --dry-run
```

Important:

- the default runtime is `openclaw`
- install will create project workspaces under `~/.malaclaw/workspaces/store/...`
- install will register OpenClaw-facing agents under `~/.openclaw/agents/...`
- install will patch `~/.openclaw/openclaw.json`

Do the real install:

```bash
malaclaw install
malaclaw doctor
malaclaw project show podcast-openclaw
```

Open OpenClaw and start with the entry-point agent:

```text
store__podcast-openclaw__content-factory__editor
```

Use a simple first prompt:

```text
Create a production package for a podcast episode about AI coding agents.
Include:
- a 5-section outline
- a short intro draft
- show notes
- 3 social post ideas
```

## 2. Understand how the demo decides what to install

For this starter, the install path is:

1. `malaclaw.yaml` includes `packs: [content-factory]`
2. the pack `content-factory` points to the team `content-factory`
3. the team template lists these agents:
   - `editor`
   - `writer`
   - `seo-specialist`
   - `social-media-manager`
   - `video-producer`
4. `editor` is the member marked `entry_point: true`

So the installation logic is:

`malaclaw.yaml` -> pack -> team -> members -> agent templates -> runtime provisioner

The `project.entry_team` field tells the user which team to open first. The real install still comes from `packs`.

## 3. Verify the project in the dashboard

From the project directory:

```bash
malaclaw dashboard
```

Open:

```text
http://localhost:3456
```

What to verify on the Overview page:

- Projects: your installed project appears
- Agents: the `content-factory` agents appear
- Skills: skill activation status is visible
- Health: inactive skills or missing requirements show up here
- Diff: install preview is visible
- Manifest: editable manifest form
- Team Graph: the team structure is visualized

What to verify on the other pages:

- Projects: inspect the installed project and team graph
- Starters: confirm the starter catalog is visible
- Config: view the current manifest and diff preview, and run install

Current dashboard customization boundary:

- project manifest edits are supported through the Overview page manifest form
- the Config page is mostly read-only preview plus install
- team topology and agent-template editing are not yet exposed as dedicated dashboard forms

## 5. Learn manifest-level customization first

This is the recommended first customization layer.

You can safely edit:

- `runtime`
- `project.id`
- `project.name`
- `project.entry_team`
- `packs`
- `skills`
- `skills[].targets`
- `project.attached_agents`

Example: make the project intent clearer and explicitly target project skills.

```yaml
version: 1
runtime: openclaw
project:
  id: podcast-openclaw
  name: Podcast OpenClaw Demo
  starter: podcast-production-pipeline
  entry_team: content-factory
  description: Podcast production demo project for learning MalaClaw
packs:
  - id: content-factory
skills:
  - id: malaclaw-cook
    targets:
      agents:
        - editor
  - id: rss-skill
    targets:
      teams:
        - content-factory
```

Then apply it:

```bash
malaclaw install --dry-run
malaclaw install
malaclaw doctor
```

## 6. Learn skill activation with a real example

The `content-factory` pack includes `last30days` as a default skill.

If you see:

```text
[INACTIVE] last30days — missing: OPENAI_API_KEY
```

that means:

- the skill exists in the pack
- the template declares it requires `OPENAI_API_KEY`
- install will skip materializing that skill until the env var exists

Activate it like this:

```bash
export OPENAI_API_KEY=sk-...
malaclaw install
malaclaw doctor
malaclaw skill check
```

For basic demo testing, it is safe to leave that skill inactive and continue.

## 7. Customize the team and topology with an overlay

This is the current path for changing:

- team members
- lead/specialist/reviewer roles
- delegation graph
- topology
- shared memory files
- agent persona/model/capabilities/skills

Create a project-local overlay:

```bash
cd /tmp/malaclaw-guided-demo/podcast-openclaw
mkdir -p ./malaclaw-templates/teams
mkdir -p ./malaclaw-templates/agents
cp /Users/zhiyuanparis/github/openclaw-dir/MalaClaw/templates/teams/content-factory.yaml ./malaclaw-templates/teams/content-factory.yaml
cp /Users/zhiyuanparis/github/openclaw-dir/MalaClaw/templates/agents/editor.yaml ./malaclaw-templates/agents/editor.yaml
```

### 7.1 Example: change topology from `star` to `lead-reviewer`

Edit `./malaclaw-templates/teams/content-factory.yaml` to this shape:

```yaml
id: content-factory
name: "Content Factory"
version: 1

communication:
  topology: lead-reviewer

members:
  - agent: editor
    role: lead
    entry_point: true
  - agent: writer
    role: specialist
  - agent: seo-specialist
    role: reviewer
  - agent: social-media-manager
    role: specialist
  - agent: video-producer
    role: specialist

graph:
  - from: editor
    to: writer
    relationship: delegates_to
  - from: editor
    to: social-media-manager
    relationship: delegates_to
  - from: editor
    to: video-producer
    relationship: delegates_to
  - from: writer
    to: seo-specialist
    relationship: requests_review
  - from: social-media-manager
    to: seo-specialist
    relationship: requests_review

shared_memory:
  dir: "~/.malaclaw/workspaces/store/<project-id>/<team-id>/shared/memory/"
  files:
    - path: pipeline-log.md
      access: append-only
      writer: "*"
    - path: content-brief.md
      access: single-writer
      writer: editor
```

Then install using the overlay:

```bash
MALACLAW_TEMPLATES=./malaclaw-templates malaclaw team show content-factory
MALACLAW_TEMPLATES=./malaclaw-templates malaclaw install --dry-run
MALACLAW_TEMPLATES=./malaclaw-templates malaclaw install
MALACLAW_TEMPLATES=./malaclaw-templates malaclaw doctor
```

### 7.2 Example: customize the lead agent

Edit `./malaclaw-templates/agents/editor.yaml` after copying it from the bundled template.

Typical edits:

- change `model.primary`
- update `soul.persona`
- tighten `soul.boundaries`
- add or remove `skills`

Then rerun:

```bash
MALACLAW_TEMPLATES=./malaclaw-templates malaclaw install
```

Important:

- overlay files replace the bundled template for that ID
- they are not partial patches
- copy the original file first, then edit it

## 8. Customize the project from the dashboard

Current best UI path:

1. Run `malaclaw dashboard`
2. Open the Overview page
3. Find the `Manifest` card
4. Edit the manifest JSON
5. Click `Save`
6. Click `Install`
7. Check `Diff`, `Skills`, `Health`, and `Projects`

Good dashboard edits today:

- change runtime
- add or remove packs
- add project skills
- retarget skills to teams or agents
- update project metadata

What still belongs in files today:

- editing team topology
- editing team graph edges
- adding brand new agents
- changing agent persona/model/capabilities in templates

Those still need the overlay directory workflow.

## 9. Recommended product improvement for dashboard customization

The current dashboard is useful for project-level manifest editing, but it is not yet the best UI for template-level customization.

The next good product step would be:

1. Add a `Customize` workflow to scaffold a project-local overlay directory.
2. Add forms for:
   - team members
   - entry-point selection
   - topology
   - agent role and model
   - skill assignment
3. Save those edits into a project-local overlay and run install from the dashboard.

That would give users the UX you are asking for without forcing every starter to copy all bundled templates up front.

## 10. Recommended testing order

If you want to learn the system cleanly, do it in this order:

1. Run the demo in `openclaw`
2. Open the dashboard and inspect the installed project
3. Edit the manifest in the dashboard and reinstall
4. Activate or ignore `last30days` intentionally so you understand skill status
5. Create a local overlay and change `content-factory` to `lead-reviewer`
6. Test the same starter in `claude-code`
7. Test the same starter in `codex`
8. Test the same starter in `clawteam`
